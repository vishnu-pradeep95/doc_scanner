Transition placement rule (CRITICAL pitfall):
- enterTransition and returnTransition → set in ENTERING fragment's onCreate()
- exitTransition and reenterTransition → set on DEPARTING fragment immediately before navigate()
- NEVER set in onViewCreated() — transitions are ignored after view creation begins

**Step 3 — PagesFragment.kt: Add onProgress callback to generatePdf() and wire it in createPdf() and createPdfFromSelection().**

Modify `generatePdf()` signature to accept an optional callback:
```kotlin
private fun generatePdf(
    ctx: android.content.Context,
    pageUris: List<Uri>,
    customBaseName: String? = null,
    onProgress: ((current: Int, total: Int) -> Unit)? = null  // ADD THIS PARAM
): File {
    val pdfDocument = PdfDocument()
    val pageWidth = 595
    val pageHeight = 842
    val total = pageUris.size   // ADD: capture total for progress

    pageUris.forEachIndexed { index, uri ->
        onProgress?.invoke(index + 1, total)   // ADD: report progress before processing each page

        // ... existing bitmap decode + page creation logic unchanged ...
    }
    // ... rest of generatePdf() unchanged ...
}
```

In `createPdf()` coroutine (the `withContext(Dispatchers.IO)` block), replace:
```kotlin
val pdfFile = withContext(Dispatchers.IO) {
    generatePdf(ctx, pages)
}
```
With:
```kotlin
// Set up progress indicator before launching IO
binding.progressIndicator.max = pages.size
binding.progressIndicator.progress = 0

val pdfFile = withContext(Dispatchers.IO) {
    generatePdf(ctx, pages, onProgress = { current, total ->
        // post() switches to UI thread from inside withContext(IO)
        binding.progressIndicator.post {
            binding.progressIndicator.progress = current
            binding.loadingText.text = getString(R.string.pdf_progress, current, total)
        }
    })
}
```

Apply the same pattern to `createPdfFromSelection()` — it also calls `generatePdf()` inside a `withContext(Dispatchers.IO)` block. Same initialization of `progressIndicator.max` and `progressIndicator.progress = 0` before the IO block, same callback wiring inside.

Do NOT use `withContext(Dispatchers.Main)` inside the callback — `View.post()` already marshals to the main thread and is safe to call from IO threads. Using `withContext(Main)` inside a `withContext(IO)` block would require nested dispatch.
