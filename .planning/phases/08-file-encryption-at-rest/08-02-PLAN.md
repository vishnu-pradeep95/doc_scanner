**PdfUtils.kt** — All 5 PDF write sites and all PdfRenderer read sites:
- Replace every `FileOutputStream(outputFile).use { fos -> *.writeTo(fos) }` with `SecureFileManager.encryptPdfToFile(outputDoc/newDoc/document, outputFile)`.
  - Line ~145 (mergePdfs): `SecureFileManager.encryptPdfToFile(outputDoc, outputFile)`
  - Line ~227 (splitPdf): `SecureFileManager.encryptPdfToFile(newDoc, outputFile)`
  - Line ~323 (extractPages): `SecureFileManager.encryptPdfToFile(newDoc, outputFile)`

- For PdfRenderer reads: where PdfUtils opens a ParcelFileDescriptor via `contentResolver.openFileDescriptor(uri, "r")`, it reads content:// URIs from the file picker or recent docs. The source PDFs may be encrypted files in app storage accessed via FileProvider URI. For file:// URIs pointing to app storage: decrypt to temp file, open PdfRenderer from temp, clean up temp in finally block. For content:// URIs: use contentResolver directly (external files are not encrypted by this app).
  - Add a helper: If the URI scheme is "file" and the file is in filesDir, use `SecureFileManager.decryptToTempFile()` then `ParcelFileDescriptor.open(tempFile, ...)`. Track temp file for cleanup.
  - If URI scheme is "content", use existing `contentResolver.openFileDescriptor()`.
- For the compressPdf JPEG intermediate (line ~408): `FileOutputStream(tempJpeg)` writes a temp file in cacheDir. Do NOT encrypt temp files in cacheDir (anti-pattern from research). The `BitmapFactory.decodeFile(tempJpeg.absolutePath)` at line ~414 reads the same temp. Leave both as-is.

- For PdfRenderer input: The `extractPages()` method opens PDFs via `contentResolver.openFileDescriptor()`. Apply same decrypt-to-temp pattern as PdfUtils for file:// URIs in app storage.

For all decrypt-to-temp patterns, ensure temp files are deleted in `finally` blocks. Use `context.cacheDir` for temp files.
